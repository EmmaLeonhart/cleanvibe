cleanvibe

A zero-dependency CLI that drops Claude Code into a well-structured project — every time.

cleanvibe is not a coding tool. It is a state initializer: it removes the friction between "I want to build something" and "Claude is working inside a disciplined, documented environment." The real payload is the CLAUDE.md behavior contract it injects — plan-into-queue-first, commit early and often, keep docs honest.

pip install cleanvibe

What is cleanvibe, and why use it?

Most projects start with blank-repo paralysis, drift into poor commit hygiene, and host an AI assistant that rambles without producing durable artifacts. cleanvibe fixes the starting conditions:

  • An opinionated CLAUDE.md contract — plan into queue.md before executing; update the queue in the same commit as the work; commit early and often; keep documentation truthful.
  • queue.md + todo.md — concrete executable steps vs. long-horizon backlog. Work flows todo.md → queue.md → git log. An interrupted session resumes from the queue, not lost chat context.
  • A data_lake/ (for new) created from the first commit, so you can drop specs, exports, and sample data straight in.
  • Zero dependencies — pure Python stdlib. Fast install, no supply chain. Works on Windows, macOS, Linux (Python 3.9+).

Three subcommands cover three situations. Pick a tab above:

cleanvibe newStart a brand-new project from nothing. cleanvibe cloneOnboard an existing codebase. cleanvibe replicateTurn an arXiv/alphaxiv paper into a replication project.

Creating a new project

cleanvibe new my-project

What happens:

  1. Creates my-project/.
  2. Writes CLAUDE.md — the AI-development workflow contract.
  3. Writes README.md — starter documentation.
  4. Writes queue.md — pre-seeded with a first-session bootstrap sequence that walks Claude through triaging dropped-in files, inferring the project, interviewing you, creating todo.md, populating the real queue, and pushing to a private GitHub repo.
  5. Writes .gitignore (sensible Python defaults).
  6. Creates data_lake/.gitkeep — the data lake exists from commit 1, so you can drop files in before the first session.
  7. Initializes a git repo with an informative initial commit.
  8. Launches Claude Code inside the project.

Options: --dry-run previews everything without writing; --no-claude skips launching Claude.

Cloning a project — codebase onboarding

cleanvibe clone https://github.com/user/repo

clone is deliberately different from new: it is for onboarding an existing codebase, not bootstrapping a blank one.

  1. git clone the repository.
  2. Create and check out a dedicated cleanvibe-onboarding branch — the default branch is left untouched.
  3. Prepend-or-write an onboarding CLAUDE.md and queue.md: if the repo already has them, the fresh block goes on top (newest first) and the original content is preserved below — re-running just layers another block.
  4. Inject .gitignore only if missing. No data_lake/ (nothing was dropped in — it is a real codebase) and no README overwrite.
  5. Commit the onboarding scaffold on the branch.
  6. Launch Claude Code.

The onboarding queue.md is small and focused: read & document the repo, make existing docs honest, rewrite CLAUDE.md to the repo's real development practices, add tests/CI if sparse, then synthesize any existing planning artifacts and hand off to the repo's own todo.md.

Replicating a paper

cleanvibe replicate https://arxiv.org/abs/2201.02177

Accepts an arXiv or alphaxiv id / abs URL / pdf URL. It scaffolds a standalone replication project named replicating-<paper-slug> (silently suffixed -2/-3 on collision — you supplied no name, so it never errors and never prompts; deliberately asymmetric with new).

Every replication produces three compounding artifacts:

  • The runnable replication — code under src/ driven by SKILL.md, the agent-executable replication plan.
  • A legibility layer — a published findings report (GitHub Pages site + transportable PDF), built by GitHub Actions.
  • The reusable SKILL.md — operationalized replication methodology that compounds across papers.

Structure highlights: the paper lives at replication_target/paper.pdf (gitignored — never in data_lake/); the authors' code is cloned as a git submodule under replication_target/; a downloadable ZIP replication package and the Pages site are produced by Actions, not committed. See docs/replication_framing.md for the full "replication infrastructure" vision.

Worked example: a real run of this pipeline on Grokking (arXiv:2201.02177) — scaffold, paper download, a faithful tiny-transformer reimplementation, and an honest FINDINGS.md.
